# Visual Brief Layout

**Purpose:** The current brief is one long wall of streamed markdown. Reps will skim, not read. This feature adds light visual structure — a key-metrics row computed from the snapshot, and section containers that mirror the playbook's hiking trail (Fires / Leverage / Slow Burns / Questions). Goal: the brief feels like a structured deliverable a rep can navigate, not a 2,500-word essay they have to plow through.

## Design Principle Alignment

- **Implements vision principle #6** (*"Earn rep attention"*): if the brief isn't worth more than the time to read it, the rep stops uploading. Structure is part of "worth more."
- **Respects vision principle #1** (*"Coach, not calculator"*): the metrics row gives top-of-page context; it does NOT replace the brief. The brief still does the coaching.
- **Respects vision principle #2** (*"The playbook is the IP"*): we don't change what the coach says, only how it's laid out after streaming completes.

## Feature: Key Metrics Row

### Scenario: Metrics row appears above the brief on every run
Given a rep uploads a territory spreadsheet with renewal dates and ARR values
When the brief generation starts
Then a metrics row is displayed at the top of the brief area showing four st.metric widgets:
  - Total accounts in territory
  - ARR at risk in Red Zone (sum of ARR for accounts renewing in next 120 days)
  - Red Zone fires (count of accounts in the next 120 days with Open Opp = N or blank)
  - Days to next renewal (smallest positive day-count across all accounts)

### Scenario: Empty Red Zone gracefully handled
Given a territory with no accounts renewing in the next 120 days
When the metrics row renders
Then "ARR at risk" shows $0
And "Red Zone fires" shows 0
And "Days to next renewal" still shows the actual smallest positive day-count

### Scenario: Missing renewal date column degrades cleanly
Given a territory upload that doesn't include a "Customer Renewal Date" column
When the metrics row tries to render
Then renewal-date-dependent metrics show "—" instead of crashing
And total accounts still renders correctly

### Scenario: ARR formatted human-readably
Given an ARR sum of 1850000
When displayed in the metrics row
Then it shows as "$1.85M" (not "$1850000.00")

## Feature: Sectioned Brief Rendering

### Scenario: Brief streams into a live container during generation
Given the brief is currently being generated by the LLM
When tokens stream back from the API
Then they accumulate in a single placeholder block in real-time (current behavior preserved)
And the structured layout is not yet rendered

### Scenario: Brief restructures into sections when streaming completes
Given the brief stream has finished
When the final response is received
Then the streaming placeholder is replaced
And each top-level section in the brief is rendered in its own `st.container(border=True)`
And section detection uses H2 markdown markers (`## ...`)

### Scenario: Today's headline renders above all sections
Given a brief that begins with intro prose before the first H2 marker
When the brief is parsed for layout
Then content before the first H2 is rendered in a distinct top container labeled "Today's headline"

### Scenario: Recognized section emojis map to canonical zones
Given a brief with sections marked `## 🔴 The Fires`, `## 🟠 The Leverage Plays`, `## 🟢 The Slow Burns`, `## What You Don't Know` (or `## What I Don't Know`)
When sections are rendered
Then each appears in its own bordered container with the original heading preserved
And ordering follows the source brief (do not re-order sections)

### Scenario: Unrecognized section names still render
Given a brief contains an H2 section the parser doesn't recognize (e.g. `## The pattern you need to diagnose`)
When sections are rendered
Then the unrecognized section gets its own bordered container with its heading preserved
And nothing is dropped or hidden

### Scenario: Brief without H2 markers falls back gracefully
Given a brief that doesn't follow the expected structure (no H2 markers at all)
When the brief is parsed for layout
Then the entire brief renders in a single container as a fallback
And no error is thrown

### Scenario: Dollar escaping still applies to structured output
Given a brief containing dollar amounts like "$1.1M" or "$610K"
When sections are rendered
Then dollar signs are escaped (`\$`) before passing to `st.markdown`
And dollar amounts display plainly (no MathJax/LaTeX rendering)

### Scenario: Run details stay below all section containers
Given the brief has been fully rendered
When the user looks at the page
Then the "Run details" expander appears below all section containers (existing behavior preserved)

## Implementation Notes

Two new pure functions added to a new module `brief_layout.py`:

```python
def compute_brief_metrics(df: pd.DataFrame, today: date | None = None) -> dict:
    """Returns metrics dict with keys: total_accounts, arr_at_risk, red_zone_fires, days_to_next_renewal.
    Each value may be None if the underlying column is missing (UI shows "—")."""

def parse_brief_sections(brief_markdown: str) -> list[dict]:
    """Splits brief on H2 markers. Returns list of {heading, body, kind} dicts.
    `kind` is one of: 'headline' (pre-first-H2 content), 'fires', 'leverage', 'slow_burns', 'questions', 'other'.
    Returns single-section list with kind='unstructured' if no H2 markers found."""
```

Rendering wired in `app.py` after stream completes:
1. Clear streaming placeholder
2. Render metrics row using `st.columns(4)` + `st.metric` per stat
3. For each section returned by `parse_brief_sections`: `with st.container(border=True): st.markdown(...)`

## Design Decisions

**Why inline accounts within Fires (not per-account expanders):**
A Streamlit `st.expander` per account would add a click-to-open step for every fire. The whole point of the brief is that the rep reads the reasoning. Hiding it behind expanders adds friction without saving real estate (the brief is already on a single scrollable page). One container per *zone* gives visual separation; per-account drill-down belongs in a future "account detail" feature, not here.

**Why parse markdown instead of asking the LLM for structured JSON:**
Asking Claude to return JSON would force two output passes (structured + prose) or sacrifice the streaming UX. Parsing on H2 boundaries is reliable because we control the system_prompt that defines the output shape. If the brief format ever drifts, the unrecognized-section scenario ensures graceful fallback.

**Why metrics computed from snapshot data, not parsed from brief:**
The brief is LLM output — variable, occasionally hallucinated, sometimes summarized. The metrics row is data — deterministic, fast, trustworthy. Reps look at it for an unambiguous "is anything on fire" gut check before reading the brief.

## Out of Scope

- **Filtering / sorting within sections.** Future feature if reps ask for it.
- **Per-account drill-down cards** with full SF row data inline. Future, possibly part of "ready to send" view (#33).
- **Export brief to PDF / email.** Future, ties into Phase 5 (push delivery).
- **Mobile responsive layout.** Streamlit handles basic responsive behavior; we're not optimizing for phones.
- **Sticky headers / scroll-spy / table of contents.** Overkill for current brief length.
- **Color theming beyond zone emojis.** Defer until Streamlit-throwaway-shell phase ends.

## Learnings

- **H2 detection wants a negative lookahead, not a string match.** `line.startswith("## ") and not line.startswith("### ")` looks correct but breaks on `#### Even deeper`. The regex `^##(?!#)\s+` rejects any line with three-or-more leading hashes in one pass. Cleaner and harder to get wrong.
- **`saw_h2` flag is load-bearing for the headline-vs-unstructured distinction.** Pre-first-H2 prose becomes a "Today's Headline" section *only when* there are subsequent H2s. If the brief has no H2s at all, the same prose should be classified `unstructured` with no heading. Without the flag, all no-H2 briefs would render with the "Today's Headline" label, which is wrong.
- **`placeholder.container()` is how you replace a Streamlit `st.empty()` with multiple elements.** Calling `.markdown()` on a placeholder swaps the single element. To put a headline + 4 sections + 4 containers in its place, use `with placeholder.container(): ...` — anything inside that block becomes the placeholder's new content as one unit.
- **Compute metrics from the snapshot, never parse them from the brief.** The brief is variable LLM output; the metrics row is data the rep relies on for an unambiguous "is anything on fire" gut check. Mixing those layers tempts you into parsing prose for numbers, which always rots. The metrics row stays trustworthy because it never touches LLM output.
- **`format_arr` trailing-zero handling: rstrip then rstrip.** `f"{2.0:.2f}".rstrip("0").rstrip(".")` → `"2"`. The double rstrip handles both `"2.00"` → `"2"` and `"1.85"` → `"1.85"` (no `.` to strip) without a conditional.
